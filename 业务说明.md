# 🚀 ChatSao — 基于 Flutter + Spring Boot 的即时通信 (IM) 系统

> **项目状态：核心功能已全部实现，可正常运行**

## 1. 项目概述

ChatSao 是一款极具未来感与人文温度的现代即时通信 App。前后端分离架构，前端使用 Flutter 构建跨平台应用（支持 Web / Android / iOS），后端使用 Spring Boot 提供 RESTful API 并通过 WebSocket 维持长连接实现实时通信。

### 1.1 核心业务需求（已全部实现）

| 功能模块 | 状态 | 说明 |
|---------|------|------|
| **用户鉴权** | ✅ 完成 | 注册、登录、JWT Token 持久化 |
| **好友管理** | ✅ 完成 | 搜索用户、添加好友、接受/拒绝请求、好友列表 |
| **实时私聊** | ✅ 完成 | 1v1 文本消息 WebSocket 实时收发 |
| **语音消息** | ✅ 完成 | 前端录音 → HTTP 上传 → WebSocket 发送 → 对端播放 |
| **离线历史** | ✅ 完成 | MySQL 持久化，支持分页拉取聊天记录 |
| **会话列表** | ✅ 完成 | 显示最后一条消息预览、时间戳、未读红点计数 |
| **已读回执** | ✅ 完成 | 进入聊天自动标记已读，清除未读数 |
| **在线状态** | ✅ 完成 | WebSocket 连接状态实时展示 |
| **Redis 缓存** | ✅ 完成 | 在线状态管理、未读消息计数缓存 |

### 1.2 UI/UX 设计规范

- **暖色渐变背景** — 奶油色系渐变底色
- **Glassmorphism（毛玻璃）** — 输入框、导航栏、卡片均采用毛玻璃效果
- **Organic Shapes（有机形状）** — 不对称圆角 Blob 形状的头像和按钮
- **微动态交互** — 录音波形动画、语音播放动画、页面切换过渡
- **底部导航栏** — 消息页 / 联系人页 / 设置页 三 Tab 切换

## 4. 数据库设计

### 4.1 MySQL 表结构

```sql
-- 用户信息表
CREATE TABLE `user_info` (
  `user_id` BIGINT AUTO_INCREMENT PRIMARY KEY,
  `username` VARCHAR(50) NOT NULL UNIQUE,
  `password` VARCHAR(255) NOT NULL,       -- BCrypt 加密存储
  `avatar_url` VARCHAR(255) DEFAULT NULL,
  `create_time` DATETIME DEFAULT CURRENT_TIMESTAMP,
  `update_time` DATETIME DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;

-- 好友关系表
CREATE TABLE `friend_relation` (
  `id` BIGINT AUTO_INCREMENT PRIMARY KEY,
  `user_id` BIGINT NOT NULL,
  `friend_id` BIGINT NOT NULL,
  `status` TINYINT DEFAULT 0 COMMENT '0-待通过, 1-已通过, 2-已拒绝',
  `create_time` DATETIME DEFAULT CURRENT_TIMESTAMP,
  UNIQUE KEY `uk_user_friend` (`user_id`, `friend_id`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;

-- 聊天记录表（核心）
CREATE TABLE `chat_message` (
  `msg_id` BIGINT AUTO_INCREMENT PRIMARY KEY,
  `sender_id` BIGINT NOT NULL,
  `receiver_id` BIGINT NOT NULL,
  `chat_type` TINYINT DEFAULT 0 COMMENT '0-私聊',
  `msg_type` TINYINT DEFAULT 0 COMMENT '0-文本, 3-语音',
  `content` TEXT NOT NULL,               -- 文本内容 或 音频 URL
  `is_read` TINYINT DEFAULT 0 COMMENT '0-未读, 1-已读',
  `send_time` DATETIME DEFAULT CURRENT_TIMESTAMP,
  INDEX `idx_sender_receiver` (`sender_id`, `receiver_id`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;
```

### 4.2 Redis 缓存设计

| Key 格式 | 类型 | 用途 | Value 示例 |
|----------|------|------|-----------|
| `user_online:{userId}` | String | 用户在线状态 | `"1"` = 在线 |
| `im:unread:{receiverId}` | Hash | 未读消息计数 | `{senderId: count}` |

## 5. RESTful API 接口清单

### 5.1 公开接口（无需鉴权）

| 方法 | 路径 | 说明 | 请求体 |
|------|------|------|--------|
| POST | `/api/user/register` | 用户注册 | `{username, password}` |
| POST | `/api/user/login` | 用户登录 | `{username, password}` → 返回 `{token, userId, username}` |

### 5.2 受保护接口（需 Bearer Token）

#### 好友模块 `/api/friend`

| 方法 | 路径 | 说明 | 参数 |
|------|------|------|------|
| GET | `/search?username=xxx` | 模糊搜索用户 | Query |
| POST | `/add` | 发送好友请求 | Body: `{friendId}` |
| GET | `/requests` | 获取我的好友请求列表 | - |
| POST | `/accept` | 接受好友请求 | Body: `{friendId}` |
| POST | `/reject` | 拒绝好友请求 | Body: `{friendId}` |
| GET | `/list` | 获取好友列表（联表查头像昵称） | - |

#### 聊天模块 `/api/chat`

| 方法 | 路径 | 说明 | 参数 |
|------|------|------|------|
| GET | `/sessions` | 获取会话列表（最后消息+未读数+msgType） | - |
| GET | `/history?friendId=&limit=&offset=` | 获取聊天历史记录 | Query |
| PUT | `/read?friendId=` | 标记与某好友的消息为已读 | Query |

#### 文件模块 `/api/file`

| 方法 | 路径 | 说明 | 参数 |
|------|------|------|------|
| POST | `/upload` | 上传音频文件 | Multipart: `file` → 返回可访问 URL |

### 5.3 WebSocket 接口

| 项目 | 值 |
|------|-----|
| **端点** | `ws://localhost:8081/ws/chat?token={jwt_token}` |
| **握手鉴权** | URL 参数中的 token 通过 `JwtHandshakeInterceptor` 校验 |

## 6. WebSocket 通信协议规范

### 6.1 客户端 → 服务端

#### 心跳保活
```json
{ "type": "PING" }
```
→ 服务端回复 `{ "type": "PONG" }`（前端不透传给 UI）

#### 发送聊天消息
```json
{
  "receiverId": 1002,
  "content": "你好",
  "msgType": 0
}
```
> `msgType`: `0`=文本, `3`=语音（语音时 content 为音频 URL）

### 6.2 服务端 → 客户端（推送）

```json
{
  "msgId": 9527,
  "senderId": 1001,
  "receiverId": 1002,
  "content": "你好",
  "msgType": 0,
  "chatType": 0,
  "sendTime": "2026-05-02T19:18:08"
}
```

> **关键设计**: 消息同时推送给发送方和接收方双方，保证发送方也能即时看到自己发出的消息。

## 7. 语音消息完整流程

```
[前端录音]                          [后端处理]
    │                                   │
    ├─ 1. 点击麦克风按钮开始录音          │
    │   (record 包, Opus/WebM for Web)   │
    │                                   │
    ├─ 2. 再次点击停止录音               │
    │   AudioRecorder.stop()            │
    │   → 返回 blob URL (Web) / 文件路径 │
    │                                   │
    ├─ 3. HTTP POST 上传音频文件         │
    │   POST /api/file/upload            │
    │   → 返回 http://.../uploads/xxx   │
    │                                   │
    ├─ 4. WebSocket 发送语音消息         │
    │   { receiverId, content: url,      │
    │     msgType: 3 }                  │
    │                                   │
    ├─ 5. 后端持久化 + 推送给对端        │
    │   saveMessage() → DB              │
    │   push to receiver session        │
    │                                   │
    └─ 6. 对端收到 → 渲染语音气泡        │
        点击播放 → audioplayers 播放     │
```

### Web 平台特殊处理

| 问题 | 解决方案 |
|------|----------|
| `AudioEncoder.opusWebM` 不存在 | 使用 `AudioEncoder.opus`（底层自动选 audio/webm;codecs=opus） |
| `dart:io.File` 不可用 | 通过 Dio GET 获取 blob URL 字节数据，用 `MultipartFile.fromBytes` 上传 |
| `hasPermission()` 可能静默失败 | Web 端跳过权限检查，由 record 包内部在 start() 时自动请求麦克风权限 |
| record.js 不需要 | record_web v1.3.0 使用 dart:js_interop 原生实现，移除外部 JS 引用 |

## 8. 性能优化与健壮性设计

### 8.1 前端优化

| 机制 | 实现 |
|------|------|
| **心跳保活** | 每 30s 发送 PING，维持 WebSocket 连接不被中间设备断开 |
| **指数退避重连** | 断线后按 3s → 6s → 12s → ... → 60s(max) 间隔重连 |
| **消息去重** | 收到新消息前根据 msgId 判断是否已存在，防止重复渲染 |
| **事件总线** | `AppEventBus`（StreamController.broadcast）替代 GlobalKey 跨组件通信 |
| **连接状态机** | `_connected` / `_connecting` 双标志位防止并发重连风暴 |

### 8.2 后端优化

| 机制 | 实现 |
|------|------|
| **Redis 缓存** | 在线状态 String 存储、未读计数 Hash 存储 |
| **会话管理** | ConcurrentHashMap<Long, WebSocketSession> 线程安全映射 |
| **ThreadLocal 用户上下文** | JwtInterceptor 解析 token 存入 UserContext，后续通过 `UserContext.getUserId()` 获取 |
| **静态资源映射** | `/uploads/**` → `file:{uploadDir}/` 支持音频文件直接访问 |
| **Redis 容错** | 所有 Redis 操作包裹 try-catch，异常不影响主流程 |
